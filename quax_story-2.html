Chapter 1 — The Data

Where our batch is born

A faint signal, buried in noise — and the one trick that pulls it out.

The Problem

Chasing a whisper of dark matter

QUAX searches for the axion, a leading candidate particle for dark matter. The detector produces an extremely faint radio signal — and it never arrives clean. Every single reading is buried under random electronic noise, so strong that one scan on its own tells you almost nothing. Our task is to turn that raw, noisy stream into a signal a physicist can read, and to do it in real time as the data keeps coming.

  • Data arrives as paired I/Q binary files — a fresh pair roughly every 5 seconds
  • Each file holds thousands of quick "scans" of the detector
  • A single scan is dominated by noise; the real signal is far too weak to see

Meet our hero: one batch

A batch is one pair of I/Q files — about 4096 scans of 2048 samples each. We'll follow this one batch through every stage of the pipeline, and then through two very different experiments.

Up next — if one scan is just noise, how do we ever see the signal? There's a beautifully simple answer.
The Idea

Averaging turns noise into signal

Here's the key insight the whole experiment rests on. The noise in each scan is random — sometimes positive, sometimes negative — so when you average many scans together, the noise tends to cancel itself out. The real signal, however, sits at the same frequency every time, so it survives the averaging and steadily grows clearer. Take an FFT of every scan, average the results, and a peak emerges from what looked like pure static.

In short: one scan = noise. Thousands of scans, averaged = a signal. The more scans we fold in, the sharper the peak becomes.

Chapter 2 — The Pipeline

How the batch travels

The architecture, the message queue, and one smart shortcut.

Architecture

A chain of small services

The pipeline is built as a set of independent stages joined by a message queue. Each stage does exactly one job and hands off to the next — which means any stage can be scaled, restarted, or completely swapped without disturbing the others. That last property is what makes our engine comparisons possible later on.

Producer

Announces new data by dropping a work order onto the queue

Kafka

Holds the queue of pending and finished work

Processor

Does the FFT — the one swappable part (Dask or Spark)

Dashboard

Plots the finished spectrum live

The processor is the only piece we change between experiments. Everything else stays fixed — same producer, same Kafka, same dashboard — so any difference we measure comes from the engine alone.
Up next — why put a queue in the middle at all? Why not connect the pieces directly?
The Message Queue

Kafka absorbs the shocks

Data arrives at a steady pace, but processing speed naturally varies — a batch might take a moment longer, or a worker might be busy. If the producer wrote straight to the processor, any hiccup on either side would stall or crash the whole chain. Kafka sits between them as a buffer: the producer drops work in at its own pace and moves on, and the processor pulls work out whenever it's ready. If one side is briefly slower, messages simply wait in line — nothing is lost.

Up next — but there's a catch: what exactly do we put on that queue?
Design Decision

Send filenames, not files

The raw data files are large. The obvious approach — pushing the actual data through Kafka — turns out to be the wrong one, and avoiding it was one of the most important choices in the design.

✗ Push the raw data

Heavy binary files would flood the broker and blow past Kafka's message-size limit. Slow, fragile, and wasteful.

✓ Push a lightweight work order

Kafka carries only a tiny JSON with the S3 filenames (~80 bytes). Each worker then downloads its own data directly from S3, in parallel — no bottleneck.

Security note: the S3 credentials are never written into the code. They're injected into the workers through their environment, so they never appear in the task graph or get committed by accident.

Chapter 3 — Engine One

Dask does the work

Spreading the computation across the whole cluster.

Engine One · Setup

Three machines, one scheduler

The whole system runs on three Ubuntu virtual machines on CloudVeneto (2 vCPU / 4GB each). One is the master — it hosts Kafka, the dashboard, and the Dask scheduler. The other two are pure compute nodes. The Dask processor connects to all three over SSH, using the master to schedule batches and the workers to actually run the FFT.

Chapter 4 — Engine Two

Rewind: what if it were Spark?

The same batch, a second engine, and a benchmark we had to fix before we could trust it.

Engine Two

The same batch, now on Spark

Let's rewind. Our batch just went through Dask — but what if Apache Spark had done the job instead? We rebuilt the processor on Spark, keeping everything else identical: the same Kafka topics, the same S3 source, and crucially the same NumPy FFT. Spark runs as a genuine Standalone cluster across the same three machines as Dask — no shortcuts.

✓ Same FFT, one task per batch

Spark schedules each whole batch as a single task, and the FFT runs once with NumPy inside it — mirroring exactly what the Dask worker does.

✗ Not splitting the FFT itself

We could have used Spark to split one FFT across cores — but that would compare algorithms, not engines. We avoided it on purpose to keep the test fair.

And to check whether "distributed" was even worth it, we added a third contender: a plain baseline with no engine at all — the function just called directly on the master.
Up next — three configurations, same batch. Time to benchmark them — the first time, we got it wrong.

The Twist

Two cracks in the story

Before shipping that result, we stress-tested it. It didn't hold.

Crack 1 — one run isn't evidence

Spark's "highest variance" claim came from one 20-batch run. A single batch spiked to 2.65s against a typical 1.3–1.7s band. Remove that one point and Spark's stdev collapses to match the others. One outlier was doing all the work of the claim.

Crack 2 — the test couldn't lose

Submit one batch, wait, submit the next — only one task is ever in flight. Dask and Spark's whole reason to exist is scheduling many tasks across workers at once. A test that never allows that was never going to find a benefit of distributing, whether or not one exists.

So the fix isn't a better statistic bolted onto the same test. It's two honest, differently-designed benchmarks — one that fixes Crack 1, one that fixes Crack 2.

Chapter 5 — Two Honest Benchmarks

Ask it two different ways

What does one batch cost? And what happens when they pile up?

Benchmark A · Method

Per-task overhead — with real repetition

Same synchronous design as before (it's still the right tool for isolating pure per-task cost, with no concurrency confound) — but run 5 independent trials of 20 batches per engine instead of 1, and report the mean and spread across trials, not one run's internal noise.

  • Each trial: start the listener, wait for it to be ready, fire 20 batches, wait for it to finish, repeat
  • 15 runs total (3 configs × 5 trials) — fully automated, no manual babysitting between trials
Gotcha we actually hit: Python fully buffers output when it's not attached to a terminal. Our "is it ready yet?" check watched a log file that wasn't being flushed — every trial silently wasted 60 extra seconds before we added python3 -u. Small bug, big lesson in trusting timestamps over feelings.

The Verdict

Two true findings, not one

Neither benchmark alone tells the whole story — which is exactly why we needed both.

Benchmark A — one batch, isolated

Engine choice barely matters for speed. Spark's overhead — slow JVM cold start, more per-batch jitter — is real and repeatable, with no offsetting benefit at this scale.

Benchmark B — a real backlog

Distribution helps — but a large share of that help is just "stop doing one thing at a time," which ordinary concurrency gets you most of the way to before a cluster is ever needed.

The honest takeaway isn't "distributed computing doesn't matter" (Benchmark A alone would have wrongly implied that) or "distributed computing obviously wins" (an unfairly-designed load test could have oversold that). It's that these are genuinely different questions, and a benchmark answers exactly the one it was built to ask.
